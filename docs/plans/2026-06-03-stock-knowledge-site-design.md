# 股票知識網站 — 設計文件

- 日期:2026-06-03
- 狀態:設計已確認(含技術形態),待轉實作計畫
- 歸檔位置:`docs/plans/2026-06-03-stock-knowledge-site-design.md`

---

## 1. 目標與動機

- **主要目標**:個人知識整理。
- **次要目標**:公開分享。
- **核心驅動**:逼自己把投資/股票知識「寫清楚」(費曼學習法)——理解的價值來自「親手把它講清楚」的過程;公開是維持輸出的逼迫感來源。
- **明確不是**:作品集展示、SEO 流量經營、做 AI 產品。(這三個都在前期討論中被排除。)

**成功標準**:不是「站好不好看」,而是「你有沒有持續在寫」。衡量習慣是否立住,例如能否穩定維持每週一篇、累積到 20–30 篇。

---

## 2. 關鍵設計原則

整個設計圍繞這個目標的兩個最大殺手在止血:

1. **發佈摩擦** — 發佈愈麻煩,你愈少發。
2. **沒有讀者** — 沒人看 = 沒有真實逼迫感。

衍生原則:

- **YAGNI**:刻意排除一切對「自己寫知識」沒幫助、只增加維護負擔的東西(見 §9)。
- **AI 當編輯,不當作者**:保住學習命脈,AI 不得代筆產內容(見 §5)。

---

## 3. 架構

- **靜態網站產生器**:**Astro Starlight**(Astro 官方文件框架)。
  - 為什麼是 Starlight 而非裸 Astro:你想要「fooish 那種側欄+文件極簡風」,Starlight 等於「VitePress 的側欄/文件美學,但跑在你熟的 Astro 上」——兩個偏好一次滿足。
  - 開箱即有、不用自己刻:側欄主題導覽(從內容結構自動長出)、站內搜尋、深淺色模式、為閱讀調好的排版、content collections 管 Markdown、可插自訂元件。
  - 效益:把 §5 的「一次性 scaffold」工作量砍掉一大半,直接服務「壓低發佈摩擦」。
  - 退路:若日後想要更像個人 blog 的版面自由度,可退回裸 Astro + content collections 自刻版型。以目前的美學目標(知識庫、按主題分類),Starlight 是更好的選擇。
- **內容形式**:Markdown 檔,放在 repo 內(Starlight content collection,`src/content/docs/`)。
- **內容邊界**:公開文章 = repo 內的 `.md`;私人投資研究(持股、估值那些)**留在 Notion**,不與站同步。
- **部署**:**Vercel**(git push → 自動部署)。本機已裝 Vercel CLI;Astro/Starlight 零設定支援、preview 部署、自訂域名都最順。

---

## 4. 資料流 / 發佈流程

```
寫 .md  →  git push  →  Vercel 自動 build & deploy  →  上線
```

- 沒有手動 build/deploy 的儀式 —— 這是「把發佈摩擦壓到最低」的工程重點,也是唯一值得用 AI 幫忙把它一次搞好的地方。

**寫作循環**:

```
你寫初稿(學習發生在這)  →  丟 AI 當編輯/反方挑毛病  →  修  →  發佈  →  分發(見 §6)
```

---

## 5. AI 的角色(界線很重要)

- **一次性**:幫你把站 scaffold 出來(Starlight 專案、版型、RSS/SEO、自動部署設定)。
- **持續**:當你的編輯 + 魔鬼代言人——挑你講不清楚的地方、戳邏輯漏洞、扮反方。
- **紅線**:AI 不產內容、不代筆。一旦 AI 寫文章,學習與逼迫感同時歸零,且把沒內化的投資內容公開出去有風險。

---

## 6. 讀者 / 逼迫感(已拍板:cross-post 路線)

- 自架站**沒有自然流量**,逼迫感必須從別處接電。
- **已選做法**:每發一篇,把連結 + 重點丟到公開場合(Threads / X),並/或 cross-post 到方格子(canonical 指回自站,借平台讀者群)。
- 因為選了 cross-post,站要把以下「分發基礎建設」一次做對(見 §11 技術細項):RSS feed、每頁自我指向 canonical、完整 OG/Twitter 分享 meta。
- **持續性提醒**:這比站本身更決定你會不會持續寫。若哪天發現自己沒在做分發,代表逼迫感斷電,需立刻補上替代來源(公開承諾、寫作夥伴、固定節奏自我問責)。

---

## 7. 內容結構(初版,可演進)

- fooish 式:主題分類(側欄)→ 每個概念一頁。
- **第一個窄主題:「基本面與技術面」** —— 在財報/估值之前,先把兩條分析路線是什麼、各看什麼、各自的盲點講清楚。先把這條窄主題寫深寫完整,再開下一個分類(財報、估值、護城河…)。不要一開始就鋪很廣——廣而淺最容易半途而廢。
- 訂一個可持續的節奏(例:每週一篇)並寫進習慣。

目錄結構(初版):

```
src/content/docs/
  index.mdx                         ← 首頁:站的定位 + 主題地圖
  基本面與技術面/
    什麼是基本面與技術面.md          ← 第一篇(深寫完整)
```

Frontmatter schema(以 Starlight/zod 驗證,初版欄位):

- `title` — 頁面標題
- `description` — 用於 SEO/OG 描述
- `pubDate` — 發佈日期(供 RSS 排序)
- `draft`(選填)— 草稿不上線
- `sidebar`(選填)— 側欄排序/標籤

---

## 8. 投資內容合規(輕量提醒)

- 寫「知識/觀念」(估值、護城河、財報怎麼看)完全安全。
- 全站頁尾固定一行「**本站為個人學習筆記,非投資建議**」——零摩擦覆蓋合規需求。
- 若談到具體個股的買賣看法,該篇再額外強調「個人分享、非投資建議」即可(避免擦到台灣「投顧」的線)。

---

## 9. 刻意不做(防過度工程)

- 資料庫、後台 CMS
- 會員 / 登入系統
- 自建留言系統(需要的話用第三方掛件)
- 站上的 AI 功能(問答、選股小幫手等)——那是「做產品」,不是這個目標
- Notion → 站的內容同步
- 前期就追求 SEO 排名
- 每篇自動生成 OG 圖(v1 先用全站一張預設圖即可,列為 later)

> 這一節是針對「容易把整理系統愈做愈複雜」的傾向設的護欄。任何想加進來的功能,先問:它對「我有沒有持續在寫」有沒有直接幫助?沒有就不做。

## 11. 技術細項(scaffold 範圍)

這些是「一次性把站做對」要落地的具體項目,實作計畫會逐步展開:

- **專案初始化**:Astro + Starlight,設定 `site` URL(供 canonical / RSS 絕對網址使用)。
- **內容集合**:`src/content/docs/`,定義 frontmatter schema(§7)。側欄按主題分類自動生成。
- **RSS**:`/rss.xml`,使用 `@astrojs/rss`,輸出最近文章供訂閱與分發。
- **SEO / 分享 meta**:每頁自我指向 canonical + 完整 OG/Twitter 標籤(標題、描述、預覽圖)。
  - canonical 方向釐清:站上每頁 canonical 指向「自己」;「canonical 指回自站」是去方格子/Medium 那邊設定原文連結,站本身要做的是成為乾淨的 canonical 來源。
- **合規元件**:全站頁尾免責聲明一行(§8)。
- **部署**:Vercel 專案連 repo,git push → 自動 build & deploy;設定 preview 部署。
- **第一篇流程**:建立「基本面與技術面」分類 + 第一篇 `.md` 的 scaffold(空殼讓你填初稿,不是 AI 代筆)。
